Phase 1 Project skeleton + auth Week 1–2
Monorepo setup
apps/web (Next.js) + apps/api (Hono Worker) + packages/db (Drizzle schema). Shared types via packages/types.
turborepopnpm workspacestsconfig
Auth flows
Better Auth on the Hono API. Email+password + GitHub OAuth. Sessions stored in Neon. Middleware guards on protected routes.
better-authhono middlewareneon
DB schema v1
users, sessions, progress, daily_quests tables. Drizzle migrations from day one — never alter prod manually.
drizzle-ormdrizzle-kitzod schemas
Deploy pipeline
CF Pages for frontend, CF Workers for API. GitHub Actions: lint → typecheck → deploy on main push.
wranglercf pagesgithub actions
Phase 2 Game engine foundation Week 3–4
Phaser 3 integration
Dynamic import Phaser inside a React wrapper component. Canvas renders in a div, React UI overlays on top via absolute positioning. Critical: Phaser and React must NOT share the same DOM update cycle.
phaser 3dynamic importuseEffect mount
Player + world
Tilemap (Tiled editor → JSON), player sprite with run/idle/attack animations, camera follow, physics with Arcade. One level fully playable.
tiledarcade physicsspritesheet
Game ↔ React bridge
Zustand store as the bridge. Phaser fires events (BOSS_ENCOUNTERED, CONCEPT_TRIGGERED) → Zustand state changes → React renders the appropriate modal/panel. One-way data flow.
zustandphaser eventsevent emitter
Phase 3 Code challenge system Week 5–6
Monaco Editor modal
Full-screen modal with Monaco, language selector, test result panel. Lazy-loaded — don't ship it until the boss fight triggers. Render inside a React portal so it sits above the Phaser canvas.
@monaco-editor/reactreact portallazy()
Code execution API
Hono route that proxies to Judge0. User submits code → Worker sends to Judge0 → polls result → returns pass/fail + stdout. Never expose Judge0 directly to the client.
judge0hono proxyzod validation
Challenge data model
challenges table: id, title, description, starter_code, test_cases (JSONB), xp_reward, boss_id. Seed 4 challenges for daily quest day 1.
drizzlejsonbdb seed
Phase 4 Progression + daily quests Week 7–8
XP + leveling engine
Pure function: calculateXP(baseXP, timeBonus, streakMultiplier). Stored in DB on challenge completion. Level thresholds defined in a config file — easy to tune.
pure fndb transactionoptimistic UI
Daily quest reset
Cloudflare Cron Trigger on the Worker: runs at 00:00 UTC, assigns 4 random bosses to each active user's daily_quests row. No third-party scheduler needed.
cf cron triggerwrangler.toml
Profile + dashboard
Next.js RSC page. Fetches user stats server-side. Shows XP bar, level, daily quest progress, streak counter. No client-side data fetching needed here.
RSCserver componentshono fetch
Phase 5 Polish + content pipeline Week 9–10
Concept cards
MDX-based concept pages triggered mid-level. Rendered in a React modal. Store read/unread state in Zustand (persist to DB on close).
mdxnext-mdx-remote
Sound + animations
Phaser audio manager for game SFX. CSS animations for UI (boss defeat, XP pop). Keep audio assets in CF R2, load on demand.
phaser audior2css keyframes
Admin challenge editor
Simple protected Next.js admin route. Form to create/edit challenges. This is your content pipeline — no CMS needed for MVP.
admin middlewarereact hook form
Phase 6 MVP launch prep Week 11–12
Leaderboard
Simple Hono endpoint: SELECT top 20 users by XP. Cache in CF KV with 5-min TTL. Don't over-engineer this — it's a read query.
cf kvcache-control
Error tracking + analytics
Sentry for errors (free tier). Plausible or CF Analytics for page views. Add to both frontend and Hono API.
sentryplausible
Rate limiting + security
Hono rate limiter middleware on /api/submit. Input sanitization via Zod on every route. CORS locked to your domain. Done.
hono/rate-limiterzodcors
MVP scope rule: phases 1–4 are the shippable MVP. Phase 5–6 make it production. Resist adding new features until phase 4 is complete and someone has actually played through a full quest loop.
